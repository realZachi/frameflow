# Governance

Frameflow uses a lightweight, maintainer-led governance model while the community is small. The goal is to make decisions transparently and expand responsibility as contributors build trust.

## Roles

### Contributors

Anyone who reports issues, improves documentation, participates in design discussion, submits code, or helps other users is a contributor.

### Reviewers

Reviewers are trusted contributors who regularly provide useful issue triage or pull request feedback. Maintainers may invite them to help review changes in areas they know well.

### Maintainers

Maintainers can merge pull requests, manage releases, moderate community spaces, and make final project decisions. They are responsible for protecting the project's quality, security, licensing, and inclusive culture.

## Decisions

- Routine fixes and small improvements are decided through pull request review.
- Significant features, new dependencies, breaking data changes, and visual-system changes should begin with a public issue.
- Maintainers seek practical consensus and explain decisions when tradeoffs are meaningful.
- If consensus cannot be reached, maintainers make the final decision based on project scope, user impact, maintenance cost, security, and long-term coherence.

Security reports, personal data, and Code of Conduct cases are handled privately.

## Becoming a maintainer

There is no contribution-count threshold. Maintainers consider:

- Sustained, constructive participation.
- Sound technical or product judgment.
- Reliable reviews and follow-through.
- Respect for contributor and user experience.
- Understanding of the project's architecture and values.

Existing maintainers invite new maintainers publicly and document the resulting access change.

## Changes to governance

Governance changes should be proposed in an issue and merged through a pull request. As the project grows, this document may be replaced with a more formal voting or working-group model.
